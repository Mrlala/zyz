### 1.2 系统概述

中译中（中文语境翻译器）是一个面向普通用户的中文语境翻译系统，核心功能是将包含职场黑话、互联网梗、专业术语、圈层表达等难以理解的中文句子，翻译成普通人能听懂的"人话"。

**产品定位：** 中文语境下的"翻译官"，解决中文世界里的沟通障碍。

**目标用户：**
- 职场新人：看不懂领导讲话、同事汇报中的黑话
- 跨圈层用户：不理解其他圈子的梗和术语
- 中老年用户：跟不上网络流行语的变化
- 内容消费者：看不懂营销文案背后的真实含义

**核心价值：**
- 词典模式：基于 1200+ 词条库，快速查词，零延迟
- 中译中模式：AI 增强翻译，整句翻译 + 潜台词分析 + 行动建议
- 热词学习：卡片式交互，每日学习，成就激励

**技术架构：**
- 前端：Uni-app（Vue3 语法），H5 移动端优先
- 后端：Python + FastAPI + SQLAlchemy ORM
- 数据库：SQLite（后续可迁移 MySQL）
- AI 服务：DeepSeek API
- 后台管理：Vue-element-admin

**系统特点：**
- 词库优先策略：先匹配本地词库，未命中再调用 AI，兼顾速度与成本
- 词条自增长：AI 翻译结果经审核/投票后归档进词库，系统越用越聪明
- 三级内容安全：低/中/高风险分级，高风险词条只做解释不鼓励传播
- 完整成就系统：称号 + 徽章 + 等级 + 排行榜，提升用户学习动力

## 3 需求

### 3.1 要求的状态和方式

本 CSCI 在以下状态和方式下运行，不同状态和方式下具有不同的需求：

#### 3.1.1 词典模式（SRS-001）

**描述：** 词典模式是系统的核心模式之一，仅基于本地词条库进行关键词匹配，不调用 AI 服务。用户输入文本后，系统识别其中的黑话/梗/术语，展示对应词条的解释。

**特点：**
- 零延迟：纯本地词库查询，响应时间 < 100ms
- 零成本：不消耗 AI API 调用额度
- 覆盖范围：依赖词条库规模，初始 1200 条，持续增长
- 适用场景：网络不稳定、快速查词、不需要整句翻译

**优先级：** P0

**合格性规定：** 演示——输入包含已知黑话的文本，验证系统在 100ms 内返回正确的词条解释。

#### 3.1.2 中译中模式（SRS-002）

**描述：** 中译中模式是系统的核心模式之一，在词条库匹配基础上增强 AI 能力，生成整句人话翻译及结构化结果（语境判断、潜台词分析、行动建议、风险提示等）。

**特点：**
- AI 增强：调用 DeepSeek API 进行整句翻译
- 结构化输出：返回 JSON 格式的完整翻译结果（7 个模块）
- 词库辅助：AI 翻译时参考词库中的已有词条解释
- 适用场景：需要理解整句含义、分析潜台词、获取行动建议

**优先级：** P0

**合格性规定：** 演示——输入包含黑话的完整句子，验证系统返回包含 7 个模块的结构化翻译结果。

#### 3.1.3 离线模式（SRS-003）

**描述：** 离线模式在无网络连接时启用，仅支持词典查询功能，不支持 AI 翻译。系统自动检测网络状态并切换。

**特点：**
- 自动检测：网络断开时自动切换到离线模式
- 功能限制：仅支持词典查询，不支持中译中模式
- 本地缓存：依赖本地缓存的词条数据
- 适用场景：网络不稳定、无网络环境

**优先级：** P2（后续迭代实现）

**合格性规定：** 测试——断开网络连接，验证系统自动切换到离线模式并仍可进行词典查询。

#### 3.1.4 在线模式（SRS-004）

**描述：** 在线模式是系统默认运行方式，联网状态下可使用全部功能，包括词典查询、AI 翻译、热词学习、成就系统等。

**特点：**
- 功能完整：所有功能模块可用
- AI 可用：可调用 DeepSeek API 进行翻译
- 数据同步：用户学习数据、收藏数据同步到服务端
- 适用场景：正常联网环境

**优先级：** P0

**合格性规定：** 演示——联网状态下验证所有功能模块正常工作。

### 3.2 CSCI 能力需求 ⭐ 核心

#### 3.2.1 翻译能力

本节定义中译中系统的核心翻译能力需求，涵盖文本输入、关键词识别、人话翻译、结果展示等全流程。

##### 3.2.1.1 文本输入（SRS-005）

**描述：** 用户可在翻译页面的输入框中输入中文文本，系统接收文本并进行翻译处理。

**输入：**
- 输入项：中文文本
- 来源：用户手动输入 / 快捷示例点击填入
- 类型：字符串
- 格式：纯文本
- 范围：1～50 字（中文字符）
- 校验：非空、不超过50字

**输出：**
- 输出项：翻译结果
- 目的地：翻译结果展示区
- 类型：结构化 JSON
- 格式：包含人话翻译、关键词解释、语境判断等7个模块

**处理逻辑：**
1. 接收用户输入的中文文本
2. 校验文本长度（1～50字）
3. 根据当前模式（词典/中译中）进入对应处理流程
4. 词典模式：直接进行词库匹配
5. 中译中模式：先词库匹配，未命中再调用 AI

**约束条件：**
- 最大输入长度：50 字
- 最小输入长度：1 字
- 响应时间：词典模式 < 100ms，中译中模式 < 5s

**异常处理：**
- 输入为空：提示"请输入要翻译的文本"
- 超过50字：提示"最多输入50字"并截断
- 输入含特殊字符：过滤非中文特殊字符

**合格性规定：** 演示——输入50字以内的中文文本，验证系统正确接收并处理；输入超过50字，验证系统提示并截断。

##### 3.2.1.2 关键词识别（SRS-006）

**描述：** 系统自动识别用户输入文本中的黑话、梗、术语等关键词，并标注其位置与类型，为后续翻译和解释提供基础。

**输入：**
- 输入项：经校验的中文文本
- 来源：文本输入环节的输出
- 类型：字符串
- 格式：纯文本
- 范围：1～50 字
- 校验：非空字符串

**输出：**
- 输出项：关键词命中列表
- 目的地：翻译结果展示区 / 关键词解释模块
- 类型：结构化数组
- 格式：[{keyword, position, type, source}]，其中 type 取值为 黑话/梗/术语/网络流行语

**处理逻辑：**
1. 对输入文本进行分词处理
2. 在本地词库中进行精确匹配和模糊匹配
3. 命中词库的关键词直接标记类型与位置
4. 未命中词库的疑似黑话，调用 AI 进行二次识别
5. 合并词库命中结果与 AI 识别结果，去重排序
6. 返回最终关键词列表（含命中来源标识：词库/AI）

**约束条件：**
- 关键词识别准确率：词库命中 ≥ 95%，AI 识别 ≥ 80%
- 单次识别关键词数量上限：10 个
- 识别响应时间：词库匹配 < 50ms，AI 识别 < 3s
- 关键词最小长度：2 字

**异常处理：**
- 未识别到任何关键词：标记为"未命中关键词"，仅进行整句翻译
- 词库查询超时：降级为直接调用 AI 识别
- AI 识别失败：仅返回词库命中结果并提示"部分识别能力不可用"

**合格性规定：** 演示——输入含黑话/梗/术语的典型文本，验证系统正确识别并标注关键词；输入纯白话文本，验证系统返回"未命中关键词"状态；测试——构造含 10 个以上关键词的文本，验证系统按上限截断并提示。

##### 3.2.1.3 人话翻译（SRS-007）

**描述：** 系统将用户输入的整句话翻译成普通人能理解的口语化表达，去除黑话色彩，保留原意。

**输入：**
- 输入项：原始中文文本 + 关键词识别结果
- 来源：文本输入环节 + 关键词识别环节
- 类型：字符串 + 结构化数组
- 格式：原始文本 + 关键词命中列表
- 范围：1～50 字
- 校验：原始文本非空

**输出：**
- 输出项：人话翻译文本
- 目的地：翻译结果展示区（原文 vs 人话对比区）
- 类型：字符串
- 格式：1～80 字口语化中文
- 范围：长度不超过 80 字

**处理逻辑：**
1. 接收原始文本与关键词命中列表
2. 若关键词全部命中词库，优先使用词库解释进行规则化翻译
3. 若存在未命中词库的关键词或整句语境复杂，调用 AI 进行整句翻译
4. AI 翻译时携带上下文：原始文本、命中关键词及其解释、语境判断结果
5. 对 AI 返回结果进行后处理：去除冗余说明、限制长度
6. 返回最终人话翻译文本

**约束条件：**
- 翻译响应时间：词库模式 < 100ms，AI 模式 < 5s
- 翻译结果长度：1～80 字
- 必须保留原意，不得增删关键信息
- 优先级：P0
- 风格：口语化、易懂、符合普通人表达习惯

**异常处理：**
- AI 调用超时：降级为纯词库规则翻译，提示"AI 翻译能力暂时不可用"
- AI 返回结果为空或格式错误：使用词库规则翻译兜底
- 翻译结果超长：截断至 80 字并加省略号
- 翻译结果与原文相同：提示"该句已是普通人表达，无需翻译"

**合格性规定：** 演示——输入典型黑话句子，验证人话翻译准确且口语化；测试——构造 20 条含黑话的测试用例，验证翻译准确率 ≥ 80%；演示——断网状态下输入文本，验证系统降级为词库模式并给出提示。

##### 3.2.1.4 关键词解释（SRS-008）

**描述：** 系统对命中的每个关键词展示其词条解释，包含释义、示例、分类、风险等级等信息，帮助用户理解关键词含义。

**输入：**
- 输入项：关键词命中列表
- 来源：关键词识别环节的输出
- 类型：结构化数组
- 格式：[{keyword, position, type, source}]
- 校验：数组非空

**输出：**
- 输出项：关键词解释卡片列表
- 目的地：翻译结果展示区（关键词解释模块）
- 类型：结构化数组
- 格式：[{keyword, definition, example, category, risk_level, related_words}]，其中 risk_level 取值为 无/低/中/高

**处理逻辑：**
1. 遍历关键词命中列表
2. 对每个关键词查询本地词库，获取完整词条信息
3. 词库命中的关键词直接返回词条信息
4. 词库未命中（AI 识别）的关键词，调用 AI 生成临时解释
5. 按命中位置顺序排列解释卡片
6. 高风险关键词添加风险标识

**约束条件：**
- 单条解释长度：释义 ≤ 100 字，示例 ≤ 50 字
- 单次展示关键词卡片数量上限：10 个
- 响应时间：词库查询 < 50ms，AI 生成 < 3s
- 必须包含字段：释义、分类、风险等级

**异常处理：**
- 词条不存在：调用 AI 生成临时解释，并标记为"AI 生成，未经审核"
- AI 生成失败：仅展示关键词原文，提示"暂无解释"
- 词条信息不完整：缺失字段以"—"占位
- 高风险词条：在卡片显著位置展示风险提示

**合格性规定：** 演示——输入含多个关键词的文本，验证每个关键词均展示完整解释卡片；测试——构造含 AI 识别关键词的文本，验证临时解释生成并标注"AI 生成"；演示——输入含高风险关键词的文本，验证风险标识正确展示。

##### 3.2.1.5 语境判断（SRS-009）

**描述：** 系统根据用户输入文本判断其所属场景（职场/直播/游戏/梗等），为翻译、潜台词分析等环节提供语境上下文。

**输入：**
- 输入项：原始中文文本 + 关键词命中列表
- 来源：文本输入环节 + 关键词识别环节
- 类型：字符串 + 结构化数组
- 格式：原始文本 + 关键词命中列表
- 校验：原始文本非空

**输出：**
- 输出项：语境判断结果
- 目的地：翻译结果展示区（语境标签）+ 内部传递给潜台词分析/行动建议模块
- 类型：结构化对象
- 格式：{scene, confidence, reason}，其中 scene 取值为 职场/直播/游戏/梗/日常/校园/其他

**处理逻辑：**
1. 接收原始文本与关键词命中列表
2. 基于关键词的分类标签进行场景投票
3. 若关键词分类不足以判断，调用 AI 进行语境判断
4. 综合关键词投票结果与 AI 判断结果，输出最终场景
5. 给出置信度（0～1）和判断依据
6. 置信度 < 0.5 时标记为"其他"场景

**约束条件：**
- 支持的场景类型：职场、直播、游戏、梗、日常、校园、其他（共7类）
- 置信度范围：0～1，保留 2 位小数
- 响应时间：词库投票 < 50ms，AI 判断 < 2s
- 优先级：P1

**异常处理：**
- 关键词分类全为空：直接调用 AI 判断
- AI 判断失败：默认返回"其他"场景，置信度 0
- 多场景置信度接近：取最高者，并在 reason 中说明冲突

**合格性规定：** 演示——输入典型职场黑话，验证场景判断为"职场"；测试——构造 7 类场景各 5 条测试用例，验证判断准确率 ≥ 80%；演示——输入含混合场景关键词的文本，验证系统给出最高置信度场景并说明依据。

##### 3.2.1.6 潜台词分析（SRS-010）

**描述：** 系统分析用户输入文本背后的真实含义，揭示字面之下的潜台词，帮助用户理解说话者的真实意图。

**输入：**
- 输入项：原始中文文本 + 关键词命中列表 + 语境判断结果
- 来源：文本输入环节 + 关键词识别环节 + 语境判断环节
- 类型：字符串 + 结构化数组 + 结构化对象
- 格式：原始文本 + 关键词列表 + {scene, confidence, reason}
- 校验：原始文本非空

##### 3.2.1.9 词库优先策略（SRS-013）

**描述：** 系统在翻译处理时优先匹配本地词库，词库未命中再调用 AI，以降低成本并提升响应速度。

**输入：**
- 输入项：原始中文文本
- 来源：文本输入环节
- 类型：字符串
- 格式：纯文本
- 校验：非空

**输出：**
- 输出项：处理来源标识
- 目的地：内部传递（用于结果展示来源标记）
- 类型：枚举
- 格式：词库命中 / 部分命中 / AI 处理 / 降级处理

**处理逻辑：**
1. 接收原始文本
2. 第一步：在本地词库中进行精确匹配（关键词级别）
3. 第二步：若关键词全部命中词库，使用词库解释进行规则化翻译，标记来源为"词库命中"
4. 第三步：若部分关键词命中，先使用词库解释，未命中部分调用 AI 补充，标记为"部分命中"
5. 第四步：若关键词全部未命中，整句调用 AI 翻译，标记为"AI 处理"
6. 第五步：AI 调用失败时，仅使用词库部分结果，标记为"降级处理"

**约束条件：**
- 词库匹配优先级：精确匹配 > 模糊匹配
- 词库匹配响应时间：< 50ms
- AI 调用仅在词库未命中或部分命中时触发
- 词库命中率目标：≥ 60%（基于初始 1200 条词条）
- 优先级：P0

**异常处理：**
- 词库查询超时（> 100ms）：跳过词库直接调用 AI
- 词库查询异常：记录日志，降级为 AI 处理
- AI 调用失败：仅返回词库命中部分的结果，并在结果区提示"部分功能降级"
- 词库与 AI 均失败：返回错误提示"翻译服务暂时不可用"

**合格性规定：** 演示——输入词库已收录的黑话，验证系统标记"词库命中"且响应 < 100ms；测试——统计 100 次翻译的来源分布，验证词库命中率 ≥ 60%；演示——断网状态下输入未命中词库的文本，验证系统降级处理并提示。

##### 3.2.1.10 翻译历史（SRS-014）

**输出：**
- 输出项：翻译历史列表
- 目的地：我的页 - 翻译历史模块
- 类型：结构化数组

1. 翻译完成后，将原始文本、翻译结果、时间戳、模式写入存储

**异常处理：**
- 本地存储写入失败：记录日志，提示"历史保存失败，不影响翻译"
- 服务端同步失败：本地保留，下次登录后重试同步

**合格性规定：** 演示——完成多次翻译后查看我的页，验证历史记录按时间倒序展示且不超过 100 条；测试——连续翻译 105 次，验证最早 5 条被自动删除；演示——断网状态下翻译，验证本地保存成功且联网后同步；演示——点击历史记录，验证文本填入输入框可重新翻译。

##### 3.2.1.12 对比式结果展示（SRS-094）

**描述：** 翻译结果采用"原文 vs 人话"对比方式呈现，让用户一秒 get 翻译价值，直观感受黑话与人话的差异。

- 来源：文本输入环节 + 人话翻译环节
- 类型：字符串 + 字符串
- 格式：原始文本 + 翻译文本

- 目的地：翻译结果展示区顶部
- 类型：UI 组件
- 格式：左右分栏或上下分栏的对比卡片（原文区 + 人话区）

**处理逻辑：**
1. 接收原始文本与人话翻译结果
2. 渲染对比卡片：上方/左侧展示原文，下方/右侧展示人话翻译
3. 原文区高亮命中关键词（使用品牌色标注）
4. 人话区使用更易读的字号与配色
5. 在两区之间添加视觉引导（如箭头、"翻译为"文案）

- 关键词高亮颜色：品牌主色
- 字号：人话区字号 ≥ 原文区字号，确保可读性
- 响应时间：渲染 < 100ms
- 优先级：P0

**异常处理：**
- 人话翻译为空：仅展示原文，提示"翻译失败，请重试"
- 翻译结果与原文相同：展示对比但标注"该句已是普通人表达"
- 关键词高亮失败：原文正常展示，不影响整体对比效果
- 移动端窄屏：自动切换为上下分栏，确保可读性

**合格性规定：** 演示——输入含黑话的文本，验证对比卡片正确展示原文与人话，关键词高亮可见；演示——点击原文中高亮的关键词，验证跳转到对应解释卡片；测试——在移动端与桌面端分别查看，验证分栏方式自适应；演示——输入纯白话文本，验证系统标注"该句已是普通人表达"。

##### 3.2.1.13 解释来源可视化（SRS-099）

**描述：** 每个关键词的解释结果后面显示来源标签，区分"内置词条库"和"AI 临时解释"，并标注置信度，让用户清楚知道解释的可靠程度，体现"词库优先、AI 辅助"的技术架构。

**输入：**
- 输入项：关键词的来源信息（词库命中 / AI 生成）
- 来源：翻译引擎返回的元数据
- 类型：枚举值 + 置信度数值
- 格式：`source: "database" | "ai_temp"`，`confidence: "high" | "medium" | "low"`

**输出：**
- 输出项：来源标签 + 置信度标签 + 分类标签
- 目的地：关键词解释卡片底部
- 类型：UI 标签组件
- 格式示例：
  ```
  颗粒度
  来源：内置词条库 | 置信度：高 | 分类：职场黑话
  
  嘉豪
  来源：AI 临时解释，未审核 | 置信度：中
  ```

**处理逻辑：**
1. 翻译引擎返回每个关键词时附带 `source` 和 `confidence` 字段
2. `source = "database"`：显示"内置词条库"，置信度根据词条审核状态显示（已审核=高，待审核=中）
3. `source = "ai_temp"`：显示"AI 临时解释，未审核"，置信度固定为中
4. 分类标签仅在 `source = "database"` 时显示
5. 来源标签用不同颜色区分（词条库=绿色，AI=橙色）

**约束条件：**
- 来源标签必须显示，不可隐藏
- 置信度分为高/中/低三档
- 来源标签渲染延迟 < 10ms（随解释卡片一起渲染）

**异常处理：**
- 来源信息缺失：默认显示"AI 临时解释"，置信度"低"
- 未知来源类型：按 AI 临时解释处理

**合格性规定：** 演示——输入包含词库命中和未命中的关键词的文本，验证两类来源标签正确区分展示；审查——检查 AI 生成的临时词条均标注"未审核"。

  ```json
      {"context": "投资圈", "meaning": "买入某资产"},
      {"context": "游戏圈", "meaning": "加入队伍"},
      {"context": "日常语境", "meaning": "真的上车（交通工具）"}
    ],
    "current_context": "直播电商",
    "current_meaning": "下单购买"
  }
  ```

**处理逻辑：**
1. 关键词命中词库后，检查词条是否有多语境标注
2. 若有多语境数据：展示全部语境含义，并根据当前句子上下文高亮"当前语境"
3. 若无多语境数据但 AI 能力可用：调用 AI 补充其他语境含义
4. 当前语境判断：根据句子中其他关键词/领域词汇推断
5. UI 展示：当前语境排第一并高亮，其余语境折叠展示

**约束条件：**
- 多语境解释最多展示 5 个语境
- 当前语境判断准确率 ≥ 70%
- AI 补充多语境响应时间增量 < 2s
- 多语境数据来源同样遵循 3.2.1.13 来源可视化规则

**异常处理：**
- 语境判断失败：按默认顺序展示，不高亮任何语境
- AI 补充失败：仅展示词库中已有的语境含义
- 词条只有单一含义：不展示多语境区域

**合格性规定：** 演示——输入"这个车要不要上"（歧义句），验证系统展示"上车"的多个语境含义；演示——输入"直播间上车"验证当前语境正确高亮为"直播电商"。

##### 3.2.1.15 风险分层展示（SRS-101）

**描述：** 对含有标签化、调侃、可能冒犯等风险的关键词，系统分层展示风险信息，包括风险等级、风险类型和具体建议，帮助用户理解使用该词的潜在风险。此需求增强原有的 SRS-012 风险提示，从简单的提示升级为分层展示。

**输入：**
- 输入项：关键词的风险元数据
- 来源：词条库风险字段 / AI 风险分析
- 类型：结构化数据
- 格式：`risk_level: "low"|"medium"|"high"`，`risk_types: string[]`，`advice: string`

**输出：**
- 输出项：风险信息卡片
- 目的地：关键词解释卡片底部
- 类型：UI 卡片组件
- 格式示例：
  ```
  ⚠ 风险提示
  风险等级：中
  风险类型：标签化 / 调侃 / 可能冒犯
  建议：不要用于评价现实中的具体个人
  ```

**处理逻辑：**
1. 词条库中每个词条包含 `risk_level`、`risk_types`、`advice` 字段
2. 低风险：不展示风险卡片
3. 中风险：展示黄色风险卡片，含风险类型和建议
4. 高风险：展示红色风险卡片，且只做解释，不进入热词推荐和排行榜
5. AI 翻译时，若原文包含敏感内容，AI 返回风险分析结果
6. 风险类型枚举：标签化、调侃、可能冒犯、地域歧视、性别相关、过时用语、其他

**约束条件：**
- 风险等级必须三选一（低/中/高）
- 风险类型可多选，最多 3 个
- 建议文案长度 10～50 字
- 高风险词条不进入推荐池和排行榜（与 SRS-035 联动）

**异常处理：**
- 风险数据缺失：默认为低风险，不展示卡片
- AI 风险分析失败：仅依赖词库风险字段
- 风险类型未知：归为"其他"

**合格性规定：** 演示——输入含中风险标签化词汇的文本，验证黄色风险卡片正确展示风险等级、类型和建议；审查——检查高风险词条是否被排除出热词推荐和排行榜。

##### 3.2.1.16 建议回复（SRS-102）

**描述：** 用户翻译后，系统不仅告诉用户"这句话什么意思"，还给出"怎么回复"的建议。针对学习工作赛道场景，尤其针对职场沟通，给出得体、专业的建议回复话术。此需求增强原有的 SRS-011 行动建议，从"行动方向"升级为"可直接使用的回复话术"。

**输入：**
- 输入项：翻译结果 + 语境判断 + 原文
- 来源：翻译引擎输出
- 类型：结构化数据
- 格式：原文 + 语境 + 翻译结果

**输出：**
- 输出项：建议回复文本
- 目的地：翻译结果卡片底部
- 类型：文本块
- 格式示例：
  ```
  💬 建议回复
  我再把需求拆成具体功能点、边界条件和验收标准，整理后同步给你确认。
  ```

**处理逻辑：**
1. AI 翻译时，在 prompt 中要求返回 `suggested_reply` 字段
2. 根据语境判断结果，生成对应场景的回复建议：
   - 职场场景：专业、得体、推进工作的回复
   - 社交场景：轻松、合群的回复
   - 其他场景：通用的理解确认回复
3. 回复建议可直接复制（长按或点击复制按钮）
4. 若 AI 未返回建议回复，根据行动建议（SRS-011）自动生成通用模板

**约束条件：**
- 建议回复长度 10～80 字
- 回复内容不得包含高风险词汇
- 职场场景回复需专业得体，不含口语化表达
- 响应时间增量 < 1s（随 AI 翻译一起返回）

**异常处理：**
- AI 未返回建议回复：显示"暂无建议回复，可参考行动建议"
- 语境为非职场/社交：不生成建议回复
- 建议回复含敏感内容：过滤后重新生成或隐藏

**合格性规定：** 演示——输入"这个需求颗粒度还不够"，验证系统返回专业的建议回复话术；演示——长按建议回复文本，验证可复制到剪贴板。

##### 3.2.1.17 解释质量反馈（SRS-103）

**描述：** 每次翻译结果下方提供三个反馈按钮，让用户评价解释质量，反馈数据用于持续优化词条库和 AI prompt。三个选项分别为："解释准确"、"解释不准"、"这个词过时了"。

**输入：**
- 输入项：用户点击的反馈类型
- 来源：用户操作
- 类型：枚举值（accurate / inaccurate / outdated）
- 格式：反馈类型 + 翻译记录ID + 用户ID（可选）

**输出：**
- 输出项：反馈成功提示 + 后续行为
- 目的地：UI 提示
- 类型：Toast 提示
- 格式：反馈后按钮变为已反馈状态（不可重复点击）

**处理逻辑：**
1. 翻译结果展示后，底部显示三个反馈按钮
2. 用户点击后：
   - "解释准确"：记录正面反馈，对应词条置信度权重 +1
   - "解释不准"：记录负面反馈，弹出文本框可选填"正确的解释是什么"，提交后进入纠错审核队列（与 SRS-020 联动）
   - "这个词过时了"：记录过时标记，词条标记为"待复审"，通知管理员
3. 反馈后按钮置灰，不可重复反馈
4. 反馈数据存入 `feedback` 表，关联翻译记录ID

**约束条件：**
- 每条翻译记录同一用户仅可反馈一次
- 反馈操作响应时间 < 200ms
- "解释不准"的文本框最多 200 字
- 反馈按钮必须在翻译结果展示后才能点击

**异常处理：**
- 未登录用户：允许反馈但不关联用户ID，使用设备ID
- 网络失败：本地缓存反馈数据，下次联网时同步
- 重复反馈：前端置灰拦截，后端返回 409 Conflict

**合格性规定：** 演示——翻译后点击三个反馈按钮，验证各自触发正确的后续行为；测试——同一翻译记录重复反馈，验证第二次被拦截。

#### 3.2.2 词典查询能力

##### 3.2.2.1 分类浏览（SRS-015）

**描述：** 支持用户按多级分类（一级 12 个，后续扩展二级、三级）浏览词条，便于用户从主题维度探索词典内容。

**输入：**
- 输入项：用户选择的分类ID（可空，空时返回一级分类列表）
- 来源：用户点击分类节点
- 类型：字符串/数字 ID
- 格式：category_id（可选）
- 校验：分类ID需存在且启用

**输出：**
- 输出项：分类下的词条列表 + 子分类列表
- 目的地：词典页 - 分类浏览区
- 类型：结构化对象
- 格式：{categories: [{id, name, level, parent_id, count}], entries: [{id, term, summary, category}]}，分页返回
- 分页大小：20 条/页

**处理逻辑：**
1. 接收用户选择的分类ID（首次进入默认返回一级分类）
2. 查询该分类下的直接子分类列表（含每个子分类的词条数）
3. 查询该分类下的词条列表（按更新时间倒序）
4. 分页返回结果，每页 20 条
5. 支持点击子分类进入下一级，记录浏览路径用于面包屑导航
6. 词条列表展示词条名称、简短摘要、所属分类标签

**约束条件：**
- 一级分类固定 12 个，后续版本扩展二级、三级
- 单次查询响应时间：< 200ms
- 分页大小：20 条/页
- 词条摘要长度：≤ 50 字
- 优先级：P1

**异常处理：**
- 分类ID不存在：返回一级分类列表并提示"分类不存在"
- 分类下无词条：展示空状态文案"该分类暂无词条"
- 查询超时（> 1s）：返回缓存数据并提示"加载较慢，已展示缓存"
- 接口异常：展示重试按钮

**合格性规定：** 演示——进入词典页，验证展示 12 个一级分类及各自词条数；演示——点击任一一级分类，验证展示其词条列表（20 条/页）和面包屑导航；演示——翻页加载更多，验证分页正确；演示——点击无词条的分类，验证空状态文案展示。

##### 3.2.2.2 关键词搜索（SRS-016）

**描述：** 支持用户输入关键词搜索词条，快速定位目标词条或相关词条。

**输入：**
- 输入项：搜索关键词
- 来源：搜索输入框
- 类型：字符串
- 格式：UTF-8 文本
- 校验：关键词非空，长度 1-30 字

**输出：**
- 输出项：匹配词条列表
- 目的地：词典页 - 搜索结果区
- 类型：结构化数组
- 格式：[{id, term, summary, category, match_type}]，按相关度排序
- 分页大小：20 条/页

**处理逻辑：**
1. 接收用户输入的关键词（支持防抖，300ms 触发）
2. 校验关键词长度（1-30 字）
3. 在词条库中进行模糊匹配（词条名称 > 解释 > 示例）
4. 按匹配度排序：词条名称完全匹配 > 词条名称包含 > 解释包含 > 示例包含
5. 高亮匹配关键词
6. 分页返回结果，每页 20 条
7. 记录搜索词用于热门搜索统计

**约束条件：**
- 关键词长度：1-30 字
- 搜索防抖时间：300ms
- 单次搜索响应时间：< 300ms
- 分页大小：20 条/页
- 最少返回结果数：无限制（可为 0）
- 优先级：P0

**异常处理：**
- 关键词为空：不触发搜索
- 关键词超过 30 字：截取前 30 字
- 无匹配结果：展示空状态"未找到相关词条，试试其他关键词？"并推荐热门搜索词
- 搜索超时（> 1s）：展示已缓存的部分结果并提示重试
- 接口异常：展示错误提示与重试按钮

**合格性规定：** 演示——输入已收录词条名称，验证返回该词条且排名首位；演示——输入词条解释中的关键词，验证返回包含该词的词条；测试——验证防抖 300ms 生效，输入过程中不重复请求；演示——输入无匹配关键词，验证空状态及热门搜索推荐；测试——验证关键词高亮正确展示。

##### 3.2.2.3 词条详情（SRS-017）

**描述：** 支持用户查看词条的完整信息，包括解释、示例、分类、风险等级、相关词条等。

**输入：**
- 输入项：词条ID
- 来源：用户点击词条列表中的词条
- 类型：字符串/数字 ID
- 格式：entry_id
- 校验：词条ID存在

**输出：**
- 输出项：词条详情对象
- 目的地：词条详情页
- 类型：结构化对象
- 格式：{id, term, explanation, examples[], categories[], risk_level, related_entries[], source, view_count}

**处理逻辑：**
1. 接收词条ID
2. 查询词条完整信息（解释、示例、分类、风险等级、相关词条）
3. 查询并附加浏览量、收藏数等统计信息
4. 记录浏览日志（用户ID、词条ID、时间戳）用于热度统计
5. 渲染详情页：
   - 顶部展示词条名称与风险等级标签
   - 中部展示解释文本（支持富文本）
   - 下方展示示例列表
   - 底部展示分类标签与相关词条推荐
6. 提供收藏、纠错、分享入口

**约束条件：**
- 详情页加载响应时间：< 300ms
- 风险等级标签颜色：高风险红色、中风险橙色、低风险绿色
- 解释文本支持富文本（粗体、列表）
- 示例数量：≤ 5 条
- 相关词条数量：≤ 6 条
- 优先级：P0

**异常处理：**
- 词条ID不存在：返回错误页并提示"词条不存在或已下架"
- 词条已被管理员下架：同上处理
- 风险等级缺失：默认展示为"未评估"
- 相关词条查询失败：详情页正常展示，相关词条区域展示"加载失败"
- 详情接口超时（> 2s）：展示骨架屏并提示重试

**合格性规定：** 演示——点击任一条目进入详情页，验证所有字段完整展示；演示——验证风险等级标签颜色正确（红/橙/绿）；演示——点击相关词条，验证跳转到对应详情页；测试——查看浏览日志，验证浏览量统计正确；演示——验证收藏/纠错/分享入口可用。

##### 3.2.2.4 相关词条推荐（SRS-018）

**描述：** 在词条详情页底部展示与当前词条同分类的相关词条，帮助用户扩展学习。

**输入：**
- 输入项：当前词条ID
- 来源：词条详情页加载时
- 类型：字符串/数字 ID
- 格式：entry_id
- 校验：词条ID存在

**输出：**
- 输出项：相关词条列表
- 目的地：词条详情页 - 相关词条模块
- 类型：结构化数组
- 格式：[{id, term, summary, category}]，按热度排序
- 数量：6 条

**处理逻辑：**
1. 接收当前词条ID
2. 查询当前词条所属分类
3. 查询同分类下其他词条，按浏览量倒序排序
4. 排除当前词条本身
5. 取前 6 条作为推荐
6. 若同分类词条不足 6 条，从相邻分类补充
7. 渲染推荐卡片，支持点击跳转

**约束条件：**
- 推荐数量：6 条
- 推荐策略：同分类优先，浏览量倒序
- 推荐响应时间：< 200ms
- 排除当前词条
- 优先级：P2

**异常处理：**
- 同分类无其他词条：从相邻分类补充，若仍无则隐藏该模块
- 推荐查询失败：隐藏相关词条模块，不影响详情页主内容
- 推荐数据不足 6 条：展示实际数量，不强制凑数
- 接口超时（> 1s）：展示骨架屏或隐藏模块

**合格性规定：** 演示——进入任一词条详情页，验证底部展示 6 条相关词条；演示——验证推荐词条均为同分类；测试——验证当前词条不在推荐列表中；演示——点击推荐词条，验证跳转到对应详情页；测试——同分类词条不足 6 条时，验证从相邻分类补充。

##### 3.2.2.5 词条收藏（SRS-019）

**描述：** 支持用户收藏常用词条，并在"我的"页面查看收藏列表，便于后续复习。

**输入：**
- 输入项：词条ID + 用户ID + 操作类型（收藏/取消）
- 来源：词条详情页收藏按钮
- 类型：结构化对象
- 格式：{entry_id, user_id, action}
- 校验：用户已登录，词条ID存在

**输出：**
- 输出项：收藏状态 + 收藏列表
- 目的地：词条详情页（收藏按钮状态） + 我的页（收藏列表）
- 类型：结构化对象
- 格式：{status: collected/uncollected, total_count}，列表：[{entry_id, term, summary, collected_at}]

**处理逻辑：**
1. 接收用户收藏请求（词条ID、用户ID、操作类型）
2. 校验用户登录状态，未登录则引导登录
3. 收藏操作：写入收藏记录（用户ID、词条ID、时间戳）
4. 取消收藏：删除对应收藏记录
5. 更新词条详情页收藏按钮状态（实心/空心）
6. 我的页查询：按收藏时间倒序返回用户收藏列表，支持分页
7. 收藏列表支持点击跳转到词条详情页

**约束条件：**
- 必须登录后才能收藏
- 单用户收藏数量上限：500 条
- 收藏操作响应时间：< 200ms
- 收藏列表分页：20 条/页
- 收藏按钮状态实时更新
- 优先级：P2

**异常处理：**
- 未登录：弹出登录引导，登录后自动完成收藏
- 收藏数超过上限：提示"收藏已达上限（500 条），请先取消部分收藏"

**合格性规定：** 演示——登录后点击收藏按钮，验证按钮变为实心并提示"已收藏"；演示——在"我的"页查看收藏列表，验证收藏的词条按时间倒序展示；演示——取消收藏，验证按钮变空心且列表中移除；测试——未登录点击收藏，验证弹出登录引导；测试——收藏超过 500 条，验证提示上限。

##### 3.2.2.6 词条纠错/补充（SRS-020）

- 类型：结构化对象

**合格性规定：** 演示——登录后在词条详情页提交纠错，验证提交成功并状态为"待审核"；演示——在"我的-我的提交"查看提交记录与状态；演示——管理员审核通过，验证词条内容更新且用户收到通知；演示——管理员驳回，验证用户收到驳回理由；测试——验证未登录、内容长度不合规、超每日上限等异常处理。

#### 3.2.3 热词学习能力

##### 3.2.3.1 每日热词推荐（SRS-021）

**描述：** 系统每天为用户推荐 10 个热词，混合热门词与随机词，按综合查询量和投票数排序，保证每日新鲜感与学习价值。

**输入：**
- 输入项：用户ID + 当日日期
- 来源：热词页加载时
- 类型：结构化对象
- 格式：{user_id, date}
- 校验：用户ID有效

**输出：**
- 输出项：每日热词列表
- 目的地：热词页 - 卡片堆栈
- 类型：结构化数组
- 格式：[{id, term, explanation, example, hot_score, vote_count}]
- 数量：10 条

**处理逻辑：**
1. 接收用户ID与当日日期
2. 检查用户当日是否已有推荐记录，若有则直接返回（保证日内一致性）
3. 若无，生成当日推荐：
   - 取热门词池：综合查询量+投票数排序，取 Top 50
   - 取随机词池：从全量词条中随机取 30 条（排除热门词池）
   - 从热门词池取 7 条，从随机词池取 3 条，组成 10 条
   - 按热度降序排列
4. 记录用户当日推荐列表（保证当日多次打开一致）
5. 每日凌晨 0:00 重新生成次日推荐
6. 返回推荐列表

**约束条件：**
- 每日推荐数量：10 条
- 热门词与随机词比例：7:3
- 排序公式：hot_score = query_count * 0.6 + vote_count * 0.4
- 日内推荐一致性：同一用户当日多次打开返回相同列表
- 次日刷新时间：00:00
- 推荐响应时间：< 500ms
- 优先级：P1

**异常处理：**
- 用户当日无推荐且生成失败：返回默认热门词 Top 10
- 词条池不足 50 条：使用全部词条
- 推荐服务超时（> 2s）：返回缓存的上次推荐
- 用户首次使用：随机生成并标记
- 推荐列表为空：提示"暂无热词，请稍后再试"

**合格性规定：** 演示——首次打开热词页，验证展示 10 个热词卡片；测试——当日多次打开，验证推荐列表一致；测试——次日 00:00 后打开，验证推荐列表更新；测试——验证热门词与随机词比例约为 7:3；演示——查看热词排序，验证按热度降序。

##### 3.2.3.2 卡片式交互（SRS-022）

**描述：** 热词采用大卡片式展示，一次聚焦一个热词，通过卡片堆栈效果引导用户专注学习。

**输入：**
- 输入项：热词列表（来自每日推荐）
- 来源：每日热词推荐接口
- 类型：结构化数组
- 格式：[{id, term, explanation, example, hot_score}]
- 校验：列表非空

**输出：**
- 输出项：卡片堆栈 UI
- 目的地：热词页 - 卡片区
- 类型：UI 组件
- 格式：大卡片（当前热词）+ 后方 2-3 张半透明卡片（预览）

**处理逻辑：**
1. 接收每日热词推荐列表
2. 渲染卡片堆栈：
   - 顶层：当前热词大卡片（占屏 70% 高度），展示词条名称、解释、示例
   - 后方：2-3 张半透明卡片，仅展示词条名称，营造堆栈感
3. 卡片支持手势交互（详见 3.2.3.3 左滑右滑）
4. 当前卡片滑出后，后方卡片上移并放大成为新顶层
5. 卡片上方展示进度指示器（如 3/10）
6. 卡片下方展示操作按钮（看懂/看不懂，与滑动等效）
7. 全部卡片滑完后展示"今日学习完成"页

**约束条件：**
- 卡片堆栈深度：3-4 张（1 张完整 + 2-3 张预览）
- 顶层卡片占屏高度：70%
- 卡片圆角：16px
- 卡片阴影：使用品牌阴影规范
- 卡片切换动画时长：300ms
- 字号：词条名称 ≥ 24px，解释 ≥ 16px
- 优先级：P0

**异常处理：**
- 热词列表为空：展示空状态"今日热词已学完，明日再来"
- 单条热词数据缺失：跳过该卡片，记录日志
- 卡片渲染失败：展示错误占位卡片，不阻塞后续卡片
- 卡片堆栈动画卡顿：降级为无动画切换

**合格性规定：** 演示——打开热词页，验证顶层卡片完整展示词条信息，后方有 2-3 张预览卡片；演示——滑动顶层卡片，验证后方卡片上移成为新顶层；演示——查看进度指示器，验证数字正确递增；测试——验证卡片切换动画时长为 300ms；演示——滑完全部卡片，验证展示"今日学习完成"页。

##### 3.2.3.3 左滑右滑（SRS-023）

**描述：** 用户通过左右滑动卡片标记热词掌握情况：右滑=能看懂（已掌握），左滑=看不懂（未掌握），是热词学习的核心交互。

**输入：**
- 输入项：用户滑动手势（方向 + 距离）
- 来源：触摸屏滑动手势
- 类型：手势事件
- 格式：{start_x, start_y, end_x, end_y, duration}
- 校验：水平滑动距离 > 阈值

**输出：**
- 输出项：学习记录 + 卡片切换动画
- 目的地：学习记录存储 + 卡片堆栈
- 类型：结构化对象 + 动画
- 格式：{entry_id, user_id, result: "mastered"|"unmastered", timestamp}

**处理逻辑：**
1. 监听卡片触摸事件，记录起始坐标 (start_x, start_y) 与时间戳
2. 触摸移动时，实时跟随手指位移渲染卡片（横向平移 + 轻微旋转）
3. 实时展示滑动方向提示：
   - 右滑：卡片右侧渐显"✓ 看懂"绿色标签
   - 左滑：卡片左侧渐显"✗ 看不懂"红色标签
4. 手指抬起时判断滑动结果：
   - 水平位移 ≥ 80px（或卡片宽度 25%）：触发完成
     - 右滑：标记为"已掌握"，卡片向右飞出
     - 左滑：标记为"未掌握"，卡片向左飞出
   - 水平位移 < 80px：卡片回弹到原位，不记录
5. 滑动距离 ≥ 80px 时，松手前卡片需有明显视觉反馈（如透明度变化）
6. 写入学习记录（词条ID、用户ID、结果、时间戳）
7. 触发下一张卡片上移动画（300ms）
8. 支持点击底部按钮等效操作（看懂/看不懂）

**约束条件：**
- 滑动阈值：水平位移 ≥ 80px 或卡片宽度 25%
- 滑动方向：仅识别水平滑动，垂直滑动不触发（避免与页面滚动冲突）
- 卡片跟随手指实时位移，松手前透明度从 1.0 渐变到 0.7
- 卡片飞出动画时长：300ms，使用 ease-out 缓动
- 卡片回弹动画时长：200ms，使用 spring 弹性动画
- 滑动旋转角度：最大 ±15°（随位移线性变化）
- 提示标签渐显：位移 > 30px 时开始显示
- 优先级：P0

**异常处理：**
- 滑动距离不足阈值：卡片回弹原位，不记录学习结果
- 误触垂直滑动：不触发卡片切换，交由页面滚动处理
- 学习记录写入失败：本地缓存记录，下次打开时重试同步
- 手势冲突（如多指触摸）：以首指为准，忽略其他手指
- 卡片动画卡顿：降级为无动画切换，保证学习记录正确

**合格性规定：** 演示——右滑卡片超过 80px，验证卡片向右飞出、记录为"已掌握"、下一张卡片上移；演示——左滑卡片超过 80px，验证卡片向左飞出、记录为"未掌握"；演示——滑动不足 80px 松手，验证卡片回弹原位、不记录；测试——验证滑动过程中卡片实时跟随手指位移与旋转；测试——验证"看懂/看不懂"提示标签在位移 > 30px 时渐显；测试——验证垂直滑动不触发卡片切换；演示——点击底部按钮，验证等效记录学习结果。

##### 3.2.3.4 学习统计（SRS-024）

**描述：** 统计用户热词学习情况，展示已学、已掌握、未掌握数量，让用户直观了解学习进度。

**输入：**
- 输入项：用户ID
- 来源：热词页 / 我的页加载时
- 类型：字符串/数字 ID
- 格式：user_id
- 校验：用户ID有效

**输出：**
- 输出项：学习统计数据
- 目的地：热词页顶部统计区 + 我的页学习模块
- 类型：结构化对象
- 格式：{total_learned, total_mastered, total_unmastered, today_learned, mastery_rate}

**处理逻辑：**
1. 接收用户ID
2. 查询用户全部学习记录（按结果分组统计）：
   - total_learned：已学总数（已掌握 + 未掌握）
   - total_mastered：已掌握数（result = "mastered"）
   - total_unmastered：未掌握数（result = "unmastered"）
   - today_learned：今日已学数
3. 计算掌握率：mastery_rate = total_mastered / total_learned * 100%
4. 渲染统计卡片：
   - 热词页顶部：今日已学 X/10 + 掌握率
   - 我的页：累计已学、已掌握、未掌握三项数据 + 掌握率进度条
5. 数据实时更新（每次滑动后刷新今日已学数）

**约束条件：**
- 统计响应时间：< 200ms
- 掌握率保留 1 位小数
- 统计数据缓存：客户端本地缓存 5 分钟，减少请求
- 进度条颜色：掌握率 ≥ 80% 绿色、50%-80% 橙色、< 50% 红色
- 优先级：P1

**异常处理：**
- 用户无学习记录：展示"已学 0、已掌握 0、未掌握 0"，掌握率 0%
- 统计查询失败：展示本地缓存数据并提示"数据可能不是最新"
- 分母为 0（total_learned = 0）：掌握率显示为 0%，不报错
- 数据不同步：下拉刷新强制重新拉取

**合格性规定：** 演示——学习若干热词后查看统计，验证已学/已掌握/未掌握数量正确；演示——验证今日已学数随滑动实时更新；测试——验证掌握率计算公式正确（已掌握/已学*100%）；演示——查看我的页统计模块，验证进度条颜色随掌握率变化；测试——无学习记录的新用户，验证展示 0 数据。

##### 3.2.3.5 热词排行榜（SRS-025）

**描述：** 提供完整的热词排行榜，按综合查询量与投票数排序，让用户了解全站热门趋势，支持用户投票。

**输入：**
- 输入项：排序维度（可选，默认综合）+ 时间范围（可选，默认全量）
- 来源：排行榜页加载
- 类型：结构化对象
- 格式：{sort_by: "comprehensive"|"query"|"vote", time_range: "all"|"week"|"month"}
- 校验：参数合法

**输出：**
- 输出项：热词排行榜列表
- 目的地：排行榜页
- 类型：结构化数组
- 格式：[{rank, entry_id, term, query_count, vote_count, hot_score}]
- 分页大小：50 条/页

**处理逻辑：**
1. 接收排序维度与时间范围参数
2. 查询词条热度数据（查询量、投票数）
3. 按选定维度排序：
   - 综合：hot_score = query_count * 0.6 + vote_count * 0.4
   - 查询量：按 query_count 降序
   - 投票数：按 vote_count 降序
4. 应用时间范围过滤（如"本周"只统计本周查询量）
5. 计算排名并分页返回（每页 50 条）
6. Top 3 特殊样式展示（金银铜牌）
7. 每个词条支持投票按钮（登录用户可投票，每词限投 1 票）
8. 用户已投票的词条按钮置灰

**约束条件：**
- 排序公式（综合）：hot_score = query_count * 0.6 + vote_count * 0.4
- 分页大小：50 条/页
- 排行榜刷新频率：每小时刷新一次缓存
- 单用户单词条投票上限：1 票
- 投票后不可撤销
- 排行榜响应时间：< 500ms
- 优先级：P2

**异常处理：**
- 排行榜为空：展示空状态"暂无排行数据"
- 排序参数非法：默认使用综合排序
- 投票未登录：弹出登录引导
- 重复投票：提示"已投过票"，按钮置灰
- 投票失败：提示重试，不计入票数
- 排行榜查询超时（> 2s）：返回缓存数据

**合格性规定：** 演示——打开排行榜，验证默认按综合排序展示 Top 50；演示——切换排序维度（查询量/投票数），验证列表重新排序；演示——登录后点击投票，验证票数 +1 且按钮置灰；测试——重复投票，验证提示"已投过票"；测试——验证 Top 3 特殊样式（金银铜牌）；演示——切换时间范围（本周/本月），验证列表更新。

##### 3.2.3.6 学习历史（SRS-026）

- 来源：学习历史页加载
- 类型：结构化对象

**输出：**

**处理逻辑：**

2. 校验用户登录状态
3. 查询用户全部学习记录，按学习时间倒序排列
4. 应用筛选条件（全部/已掌握/未掌握）
5. 分页返回结果（每页 20 条）
6. 渲染历史列表：每条展示词条名称、简短解释、学习结果标签、学习时间
7. 支持点击词条跳转到词条详情页
8. 支持按结果筛选切换（全部/已掌握/未掌握）
9. 支持下拉刷新与上拉加载更多

**约束条件：**
- 必须登录后才能查看
- 历史记录保留时长：永久（除非用户删除）
- 分页大小：20 条/页
- 列表按学习时间倒序
- 响应时间：< 300ms
- 单用户历史记录上限：10000 条（超出后自动清理最早的）
- 优先级：P2

**异常处理：**
- 未登录：弹出登录引导
- 无学习记录：展示空状态"还没有学习记录，去热词页学几个吧"并提供入口
- 筛选结果为空：展示"该筛选条件下无记录"
- 查询失败：展示重试按钮
- 历史记录超 10000 条：自动清理最早记录，保留最近 10000 条

**合格性规定：** 演示——学习若干热词后进入学习历史页，验证记录按时间倒序展示；演示——切换筛选条件（全部/已掌握/未掌握），验证列表正确过滤；演示——点击历史记录中的词条，验证跳转到词条详情页；测试——上拉加载更多，验证分页正确（20 条/页）；演示——无学习记录的新用户，验证空状态与入口引导；测试——历史记录超 10000 条，验证自动清理最早记录。

#### 3.2.4 用户管理能力

本节定义中译中系统的用户管理能力需求，涵盖用户登录、学习数据保存、偏好设置及本地缓存等用户侧能力。

##### 3.2.4.1 多种登录方式（SRS-027）

**描述：** 系统集成 FastAPI Users 框架，支持多种登录方式（账号密码、OAuth 第三方登录等），并预置若干测试用户供演示与体验，降低用户上手门槛。

**输入：**
- 输入项：登录凭证
- 来源：登录页表单 / 第三方登录授权回调
- 类型：结构化对象
- 格式：{login_type: "password"|"oauth", username, password, oauth_provider, oauth_code}
- 校验：账号非空、密码长度 6-32 位、OAuth 授权码有效

**输出：**
- 输出项：登录态 Token + 用户信息
- 目的地：客户端本地存储 + 全局用户状态
- 类型：结构化对象
- 格式：{access_token, token_type, expires_in, user: {id, username, nickname, avatar, role}}

**处理逻辑：**
1. 接收用户登录请求，识别登录方式（密码 / OAuth）
2. 密码登录：调用 FastAPI Users 的认证接口校验账号密码
3. OAuth 登录：回调授权码换取用户信息，首次登录自动创建账号
4. 校验通过后颁发 JWT Token，设置有效期
5. 返回 Token 与用户基本信息（昵称、头像、角色）
6. 客户端将 Token 写入本地存储，后续请求携带
7. 预置测试用户：admin/123456、test01/123456 等，方便演示

**约束条件：**
- 登录方式：账号密码 + 至少 1 种 OAuth（如微信 / GitHub）
- Token 有效期：access_token 2 小时，refresh_token 7 天
- 密码存储：使用 bcrypt 加盐哈希，禁止明文存储
- 密码强度：长度 6-32 位，包含字母与数字
- 单用户连续登录失败 5 次锁定账号 15 分钟
- 预置测试用户不少于 3 个
- 优先级：P1

**异常处理：**
- 账号或密码错误：提示"账号或密码错误"，不计入失败次数提示
- 账号被锁定：提示"账号已锁定，请 15 分钟后重试"
- OAuth 授权失败：提示"第三方登录失败，请重试或使用账号密码登录"
- Token 过期：客户端使用 refresh_token 静默续期，续期失败则跳转登录页
- 网络异常：提示"网络异常，请检查后重试"

**合格性规定：** 演示——使用预置测试用户登录，验证返回 Token 与用户信息；演示——使用错误密码登录 5 次，验证账号被锁定 15 分钟；演示——通过 OAuth 登录，验证自动创建账号并登录成功；测试——验证 Token 过期后 refresh_token 续期机制。

##### 3.2.4.2 学习数据保存（SRS-028）

**描述：** 系统保存用户的学习进度、收藏数据及翻译历史，支持跨设备同步，用户切换设备登录后数据不丢失。

**输入：**
- 输入项：学习记录 / 收藏操作 / 翻译历史
- 来源：热词学习页 / 词条详情页 / 翻译页
- 类型：结构化对象
- 格式：{user_id, type: "learn"|"favorite"|"history", payload, timestamp, client_id}
- 校验：用户已登录、payload 非空

**输出：**
- 输出项：同步状态 + 服务端最新数据
- 目的地：客户端本地存储 + 我的页
- 类型：结构化对象
- 格式：{sync_status: "success"|"conflict"|"failed", server_data, synced_at}

**处理逻辑：**
1. 用户操作产生学习数据时，先写入本地存储（保证离线可用）
2. 若用户已登录且联网，异步同步到服务端
3. 服务端接收数据，按 user_id 维度持久化到数据库
4. 多设备同步：客户端拉取服务端最新数据，按时间戳合并
5. 冲突处理：以服务端时间为准，客户端数据覆盖到服务端最新版本
6. 我的页加载时，优先展示本地数据，同时拉取服务端数据合并
7. 学习进度按词条去重，保留最近一次学习结果

**约束条件：**
- 必须登录后才支持跨设备同步（未登录仅本地保存）
- 同步频率：操作后立即同步 + 进入我的页时全量同步
- 单用户数据量上限：学习记录 10000 条、收藏 500 条、历史 100 条
- 同步响应时间：< 1s（含网络）
- 数据存储采用本地优先策略，离线可用
- 优先级：P1

**异常处理：**
- 网络异常：本地保留数据，标记为"待同步"，下次联网自动重试
- 服务端同步失败：本地保留，3 次重试后停止，提示"数据同步失败，已本地保存"
- 多设备冲突：以服务端数据为准，覆盖客户端旧数据
- 本地存储写入失败：提示"本地存储已满，请清理缓存"
- 服务端数据损坏：回滚到上次同步的快照

**合格性规定：** 演示——学习若干热词后切换设备登录，验证学习进度同步展示；演示——断网状态下学习热词，验证本地保存成功且联网后自动同步；测试——在两台设备同时操作同一收藏，验证冲突解决策略；演示——查看我的页学习记录、收藏、历史，验证数据完整。

##### 3.2.4.3 偏好设置（SRS-029）

**描述：** 系统支持用户设置个人偏好，包括默认翻译模式、风险提示开关、深色模式、字体大小等，配置项跨设备同步。

**输入：**
- 输入项：偏好设置项 + 用户ID
- 来源：我的页 - 设置入口
- 类型：结构化对象
- 格式：{user_id, settings: {default_mode, risk_warning, theme, font_size, ...}}
- 校验：设置项取值合法

**输出：**
- 输出项：设置生效状态 + 更新后的设置对象
- 目的地：客户端全局状态 + 服务端持久化
- 类型：结构化对象
- 格式：{status: "success"|"failed", settings}

**处理逻辑：**
1. 接收用户偏好设置变更请求
2. 校验各设置项取值合法性（如 default_mode 取值 词典/中译中）
3. 写入客户端本地存储，立即生效
4. 用户登录状态下，异步同步到服务端持久化
5. 应用设置：
   - default_mode：进入翻译页时默认选中该模式
   - risk_warning：控制风险提示模块是否展示
   - theme：切换深色/浅色模式
   - font_size：调整全局字号
6. 我的页加载时，读取本地 + 服务端设置，按时间戳合并

**约束条件：**
- 设置项变更立即生效，无需重启
- 设置项跨设备同步（登录用户）
- 支持的设置项：默认模式、风险提示开关、深色模式、字体大小、引导重置
- 字体大小档位：小/中/大（3 档）
- 设置同步响应时间：< 500ms
- 优先级：P2

**异常处理：**
- 设置项取值非法：忽略非法值，保留原值并提示"设置项不合法"
- 服务端同步失败：本地保留，下次联网重试
- 本地存储写入失败：提示"设置保存失败，请重试"
- 设置冲突（多设备）：以最新修改时间为准
- 重置引导：将"已引导"标记置为 false，下次进入翻译页重新展示引导

**合格性规定：** 演示——修改默认模式为"中译中"，验证进入翻译页默认选中中译中模式；演示——关闭风险提示开关，验证翻译结果区不再展示风险提示；演示——切换深色模式，验证全站配色切换；演示——在 A 设备修改设置后，B 设备登录验证设置同步生效；测试——验证字体大小 3 档切换正确。

##### 3.2.4.4 本地缓存（SRS-030）

**描述：** 系统将词条数据缓存到本地，用户在离线或网络不稳定时仍可使用词典查询等核心功能。

**输入：**
- 输入项：词条查询请求（关键词 / 分类ID / 词条ID）
- 来源：词典页 / 翻译页
- 类型：结构化对象
- 格式：{query_type: "keyword"|"category"|"entry", value, user_id}
- 校验：查询参数非空

**输出：**
- 输出项：词条查询结果（来自本地缓存或服务端）
- 目的地：词典页 / 翻译页结果区
- 类型：结构化数组
- 格式：[{id, term, explanation, examples, category, risk_level, cached_at}]

**处理逻辑：**
1. 接收词条查询请求
2. 优先查询本地缓存（IndexedDB / localStorage）
3. 本地缓存命中且未过期（TTL 24 小时）：
   - 直接返回缓存数据
   - 后台静默拉取服务端最新数据，更新缓存
4. 本地缓存未命中或已过期：
   - 联网时拉取服务端数据，返回并写入缓存
   - 离线时返回缓存数据（即使已过期）并提示"展示为缓存数据"
5. 缓存策略：LRU 淘汰，单设备缓存上限 5MB
6. 高风险词条同样缓存，但展示时强制提示风险
7. 翻译历史、收藏等用户数据也走本地缓存

**约束条件：**
- 缓存存储：IndexedDB（主）+ localStorage（辅）
- 缓存容量上限：5MB（超出按 LRU 淘汰）
- 缓存 TTL：24 小时
- 缓存命中响应时间：< 50ms
- 离线可用功能：词典查询、翻译历史查看、收藏查看
- 离线不可用功能：AI 翻译、用户提交、排行榜
- 优先级：P2

**异常处理：**
- 本地缓存不可用（如隐私模式）：直接请求服务端，提示"缓存不可用，需要联网"
- 缓存数据损坏：清空损坏数据，重新拉取服务端
- 离线且缓存未命中：提示"离线模式暂无该词条数据，请联网后查看"
- 缓存空间不足：按 LRU 淘汰最久未访问的数据
- 服务端拉取失败：使用过期缓存并标注"数据可能不是最新"

**合格性规定：** 演示——联网状态下查询词条，验证响应 < 50ms（命中缓存）；演示——断网后查询已缓存词条，验证离线可用；演示——断网后查询未缓存词条，验证提示"离线模式暂无该词条数据"；测试——验证缓存超 5MB 时按 LRU 淘汰；测试——验证缓存 24 小时后过期重新拉取。

#### 3.2.5 后台管理能力

本节定义中译中系统的后台管理能力需求，基于 Vue-element-admin 框架实现，涵盖词条管理、分类管理、用户提交审核、数据统计及风险词条标记等管理员操作功能。

##### 3.2.5.1 词条 CRUD（SRS-031）

**描述：** 管理员可在后台管理界面执行词条的增、删、改、查操作，维护词条库的完整性与准确性。

**输入：**
- 输入项：词条数据（增/改）/ 词条ID（删/查）/ 查询条件
- 来源：后台管理 - 词条管理页
- 类型：结构化对象
- 格式：{id, term, explanation, examples[], categories[], risk_level, source, status}
- 校验：管理员权限、词条名称非空、解释长度 10-2000 字

**输出：**
- 输出项：操作结果 + 词条列表
- 目的地：后台管理界面
- 类型：结构化对象
- 格式：{status: "success"|"failed", message, entry_list: [{id, term, summary, ...}]}
- 分页大小：20 条/页

**处理逻辑：**
1. 管理员登录后台，进入"词条管理"菜单
2. 查询：支持按名称模糊搜索、按分类筛选、按风险等级筛选、按状态筛选，分页展示词条列表
3. 新增：点击"新增词条"按钮，填写词条名称、解释、示例（≤5 条）、分类、风险等级，提交后写入数据库，状态为"已发布"
4. 编辑：点击列表中的"编辑"按钮，修改字段后保存，记录修改日志（修改人、时间、字段对比）
5. 删除：点击"删除"按钮，二次确认后软删除（标记 status=deleted），保留数据可恢复
6. 批量操作：支持批量删除、批量修改分类、批量标记风险等级
7. 操作日志：所有增删改操作记录到审计日志

**约束条件：**
- 仅管理员角色可访问后台词条管理
- 词条名称长度：1-30 字
- 词条解释长度：10-2000 字
- 示例数量：≤ 5 条
- 单条示例长度：≤ 50 字
- 删除采用软删除，30 天后自动物理删除
- 列表查询响应时间：< 500ms
- 优先级：P1

**异常处理：**
- 非管理员访问：返回 403 错误，提示"无权限"
- 词条名称重复：提示"词条已存在，请确认"
- 解释长度不合规：实时提示"解释长度需 10-2000 字"
- 删除已被引用的词条（如收藏/学习记录）：弹窗提示影响范围，二次确认后软删除
- 批量操作部分失败：返回失败列表，成功部分正常执行

**合格性规定：** 演示——管理员登录后台，新增一条词条，验证前台词典页可查询到；演示——编辑词条解释，验证前台详情页内容更新；演示——软删除词条，验证前台不再展示但数据库仍保留；演示——按分类与风险等级筛选词条列表，验证结果正确；测试——验证非管理员访问后台被拦截。

##### 3.2.5.2 分类管理（SRS-032）

**描述：** 管理员可在后台管理词条分类，支持多级分类层级（一级、二级、三级），维护分类树结构。

**输入：**
- 输入项：分类数据（增/改）/ 分类ID（删/查）/ 父分类ID
- 来源：后台管理 - 分类管理页
- 类型：结构化对象
- 格式：{id, name, level, parent_id, sort_order, status, description}
- 校验：管理员权限、分类名称非空、层级 ≤ 3、无循环引用

**输出：**
- 输出项：操作结果 + 分类树
- 目的地：后台管理界面 + 前台词典页
- 类型：结构化对象
- 格式：{status, message, category_tree: [{id, name, level, children: [...]}]}

**处理逻辑：**
1. 管理员进入"分类管理"菜单，默认展示分类树（折叠展开）
2. 新增分类：选择父分类（可为空表示一级分类），填写名称、描述、排序，提交后写入数据库
3. 编辑分类：修改名称、描述、排序，不影响已有词条归属
4. 删除分类：
   - 分类下有子分类：禁止删除，提示"请先删除子分类"
   - 分类下有词条：弹窗提示"该分类下有 N 条词条，请先迁移"
   - 空分类：二次确认后软删除
5. 移动分类：支持拖拽改变层级与排序，自动校验循环引用
6. 分类树查询：递归返回完整分类树，含每个分类的词条数

**约束条件：**
- 分类层级上限：3 级（一级 / 二级 / 三级）
- 初始一级分类：12 个（职场黑话、电商直播、游戏电竞、网络热梗、学术术语、体制公文、废话文学、金融圈、医疗健康、法律法务、饭圈文化、互联网黑话）
- 分类名称长度：1-20 字
- 同级分类名称不可重复
- 分类树查询响应时间：< 300ms
- 优先级：P2

**异常处理：**
- 非管理员访问：返回 403 错误
- 分类名称重复：提示"同级已存在同名分类"
- 层级超过 3 级：提示"最多支持 3 级分类"
- 循环引用：检测到循环时拒绝操作并提示"不能将分类移动到其子分类下"
- 删除非空分类：提示词条数量，要求先迁移

**合格性规定：** 演示——管理员新增二级分类，验证前台词典页可看到该分类；演示——拖拽调整分类层级，验证分类树更新且无循环引用；演示——删除空分类，验证分类树更新；测试——尝试删除有子分类的分类，验证提示"请先删除子分类"；测试——尝试删除有词条的分类，验证提示先迁移；演示——验证初始 12 个一级分类正确展示。

##### 3.2.5.3 用户提交审核（SRS-033）

**描述：** 系统提供用户提交审核功能，采用"管理员审核 + 用户投票双轨制"，简单审核流为：待审核 → 通过/拒绝 → 已发布。用户提交的新词条或纠错经审核或投票达标后归档进词库。

**输入：**
- 输入项：用户提交记录 ID + 审核操作 / 投票操作
- 来源：后台管理 - 提交审核页 / 前台投票按钮
- 类型：结构化对象
- 格式：{submission_id, action: "approve"|"reject"|"vote", reviewer_id, reason, vote_count}
- 校验：管理员权限（审核）/ 登录用户（投票）、提交记录存在

**输出：**
- 输出项：审核结果 + 提交状态变更
- 目的地：提交记录状态 + 用户通知 + 词条库
- 类型：结构化对象
- 格式：{status: "success"|"failed", submission_status: "pending"|"approved"|"rejected"|"published", message}

**处理逻辑：**
1. 用户在词条详情页提交纠错/补充，或在翻译页提交新词条，状态为"待审核"（pending）
2. 管理员审核流程：
   - 进入"提交审核"菜单，查看待审核列表
   - 点击查看详情，对比原文与提交内容
   - 通过：将提交内容合并到词条库，状态置为"已通过"（approved）→ 自动转为"已发布"（published）
   - 拒绝：填写驳回理由，状态置为"已拒绝"（rejected）
   - 通知用户审核结果（站内消息）
3. 用户投票流程（双轨制）：
   - 已发布的新词条纠错可被其他用户投票
   - 单条提交累计投票数 ≥ 10 票，自动通过审核
   - 单条提交累计反对数 ≥ 10 票，自动拒绝
   - 用户投票后不可撤销
4. 审核状态流转：pending → approved/rejected → published
5. 审核记录全部留痕，可追溯

**约束条件：**
- 管理员审核与用户投票并行（双轨制）
- 投票自动通过阈值：≥ 10 票赞成
- 投票自动拒绝阈值：≥ 10 票反对
- 单用户对同一提交限投 1 票
- 管理员审核优先级高于投票（管理员可直接终审）
- 审核时效目标：3 个工作日内
- 优先级：P2

**异常处理：**
- 非管理员尝试审核：返回 403 错误
- 重复审核同一提交：提示"该提交已审核"
- 提交内容不合规（含敏感词）：管理员拒绝并标注原因
- 投票数与审核结果冲突：以管理员审核为准
- 用户撤回提交：若已进入审核流程则不允许撤回
- 通知发送失败：记录日志，不影响审核状态变更

**合格性规定：** 演示——用户提交纠错，验证后台待审核列表出现该记录；演示——管理员审核通过，验证词条库更新且用户收到通知；演示——管理员驳回，验证用户收到驳回理由；演示——用户投票累计 10 票，验证自动通过审核；测试——验证单用户重复投票被拦截；测试——验证审核状态流转正确（pending → approved → published）。

**合格性规定：** 演示——学习 1 个热词，验证经验值 +5；演示——累计经验达 100，验证升级到 Lv2 并弹窗通知；演示——查看我的页，验证等级、总经验、进度条正确展示；测试——验证不同行为获得不同经验值；测试——验证等级仅升不降（删除记录不扣经验）；测试——验证达到 Lv10 后经验继续累加但等级不再提升。

##### 3.2.6.4 成就排行榜（SRS-091）

**描述：** 系统提供成就点数排行榜，按用户成就点数（称号+徽章+等级综合得分）排序，激发用户竞争。

**输入：**
- 输入项：排行榜维度 + 时间范围
- 来源：排行榜页加载
- 类型：结构化对象
- 格式：{sort_by: "achievement_points"|"level"|"learned_count", time_range: "all"|"week"|"month"}
- 校验：参数合法

**输出：**
- 输出项：成就排行榜列表
- 目的地：排行榜页
- 类型：结构化数组
- 格式：[{rank, user_id, nickname, avatar, achievement_points, title, level, badges_count}]
- 分页大小：50 条/页

**处理逻辑：**
1. 接收排行榜维度与时间范围参数
2. 计算用户成就点数：
   - 称号得分：小白=10、达人=50、大师=100
   - 徽章得分：每个徽章 20 分
   - 等级得分：level * 30
   - 总分 = 称号得分 + 徽章得分 + 等级得分
3. 按选定维度排序（成就点数 / 等级 / 学习数量）
4. 应用时间范围过滤（如"本周"只统计本周活跃用户）
5. 计算排名并分页返回（每页 50 条）
6. Top 3 特殊样式展示（金银铜牌）
7. 当前用户高亮展示（即使不在 Top 50，也单独展示其排名）
8. 排行榜每小时刷新一次缓存

**约束条件：**
- 排序维度：成就点数 / 等级 / 学习数量（3 种）
- 时间范围：全量 / 本周 / 本月
- 成就点数公式：称号得分（10/50/100）+ 徽章得分（20/个）+ 等级得分（level×30）
- 分页大小：50 条/页
- 排行榜缓存刷新：每小时一次
- 单用户上榜条件：成就点数 > 0
- 排行榜响应时间：< 500ms
- 优先级：P2

**异常处理：**
- 排行榜为空：展示空状态"暂无排行数据"
- 排序参数非法：默认使用成就点数排序
- 用户未上榜：单独展示其排名与成就点数
- 查询超时（> 2s）：返回缓存数据
- 成就点数计算异常：以数据库实际数据为准重新计算

**合格性规定：** 演示——打开排行榜，验证默认按成就点数排序展示 Top 50；演示——切换排序维度（等级/学习数量），验证列表重新排序；演示——验证 Top 3 特殊样式（金银铜牌）；演示——验证当前用户高亮展示；测试——验证成就点数计算公式正确；测试——验证排行榜每小时刷新缓存。

##### 3.2.6.5 成就展示（SRS-092）

**描述：** 用户可在"我的"页展示已获得的称号、徽章墙、等级及进度，直观呈现个人学习成就。

**输入：**
- 输入项：用户ID
- 来源：我的页加载
- 类型：字符串/数字 ID
- 格式：user_id
- 校验：用户ID有效

**输出：**
- 输出项：成就展示数据
- 目的地：我的页 - 成就展示模块
- 类型：结构化对象
- 格式：{user_id, current_title, level, total_exp, exp_to_next, badges: [{id, name, icon, unlocked, unlocked_at}], achievement_points, rank}

**处理逻辑：**
1. 我的页加载时，请求用户成就数据接口
2. 服务端聚合用户称号、等级、徽章、成就点数、排名数据
3. 返回结构化成就展示数据
4. 前端渲染成就展示模块：
   - 顶部：当前称号（大字号）+ 等级（Lv X）+ 成就点数
   - 中部：等级进度条（当前经验 / 下一级所需经验）
   - 下方：徽章墙（已解锁亮色 + 未解锁灰色，hover 展示解锁条件）
   - 底部：全站排名（如"全站第 128 名"）
5. 点击徽章查看解锁详情（解锁时间、解锁条件）
6. 点击称号查看称号解锁规则
7. 数据缓存 5 分钟，减少请求

**约束条件：**
- 成就展示必须位于我的页显著位置
- 徽章墙展示全部徽章（已解锁 + 未解锁）
- 已解锁徽章亮色，未解锁徽章灰色 + 解锁条件
- 等级进度条展示当前经验与下一级所需经验
- 数据缓存：客户端 5 分钟
- 成就数据加载响应时间：< 300ms
- 优先级：P1

**异常处理：**
- 用户无任何成就：展示"黑话小白"称号 + Lv1 + 0 经验 + 全部灰色徽章
- 成就数据查询失败：展示骨架屏并提示重试
- 徽章数据缺失：已解锁徽章正常展示，缺失部分不显示
- 排名计算失败：隐藏排名模块，不影响其他成就展示
- 缓存数据过期：重新拉取服务端数据

**合格性规定：** 演示——登录后查看我的页，验证称号、等级、徽章墙、进度条完整展示；演示——学习若干热词后刷新我的页，验证等级进度条更新；演示——点击徽章，验证展示解锁详情；演示——点击称号，验证展示称号解锁规则；测试——新用户查看我的页，验证展示"黑话小白"+Lv1+全部灰色徽章；测试——验证数据缓存 5 分钟生效。

##### 3.3.1.9 SRS-096 交互动效规范

##### 3.3.1.10 SRS-097 异常状态设计

- **接口描述**：为空状态、加载中、网络错误、AI 超时等异常场景定义统一的友好提示组件，避免白屏与生硬报错。

- **交互方式**：根据接口返回/异常自动切换状态；提供"重试/刷新"操作。
- **页面间跳转**：不适用（组件层）。

#### 3.3.2 AI 服务接口

##### 3.3.2.1 SRS-043 翻译接口（调用 DeepSeek API）

- **接口标识**：SRS-043，翻译接口，优先级 P0，状态已确认，使用 DeepSeek。
- **接口描述**：后端作为客户端调用 DeepSeek Chat API，将用户输入的"黑话/梗/术语"翻译为通俗易懂的"人话"，并要求模型以 JSON 结构化输出。
- **接口图（文字描述）**：客户端 → 后端 `/api/translate` → DeepSeek API `https://api.deepseek.com/v1/chat/completions` → 返回 JSON → 后端校验/补全 → 返回前端。
- **请求 URL**：`https://api.deepseek.com/v1/chat/completions`
- **请求方法**：POST
- **请求头**：
  ```
  Content-Type: application/json
  Authorization: Bearer {DEEPSEEK_API_KEY}
  ```
- **请求体（JSON 示例）**：
  ```json
  {
    "model": "deepseek-chat",
    "messages": [
      {
        "role": "system",
        "content": "你是"中译中"翻译助手，将包含黑话、梗、术语的中文翻译成普通人能理解的人话。请严格以 JSON 对象返回，包含字段：translation、keywords、context、subtext、suggestion、risk、related。risk 取值 low/medium/high。"
      },
      {
        "role": "user",
        "content": "翻译：赋能"
      }
    ],
    "response_format": { "type": "json_object" },
    "temperature": 0.7,
    "max_tokens": 1024
  }
  ```
- **响应体（JSON 示例）**：结构见 3.3.2.2，DeepSeek 原始响应包络如下：
  ```json
  {
    "id": "chatcmpl-xxx",
    "object": "chat.completion",
    "model": "deepseek-chat",
    "choices": [
      {
        "index": 0,
        "message": {
          "role": "assistant",
          "content": "{\"translation\":\"...\",\"keywords\":[],\"context\":\"\",\"subtext\":\"\",\"suggestion\":\"\",\"risk\":\"low\",\"related\":[]}"
        },
        "finish_reason": "stop"
      }
    ],
    "usage": { "prompt_tokens": 0, "completion_tokens": 0, "total_tokens": 0 }
  }
  ```
- **错误码定义**：

  | HTTP 状态码 | 含义 | 处理策略 |
  |---|---|---|
  | 200 | 成功 | 解析 `choices[0].message.content` 为 JSON |
  | 401 | 鉴权失败（API Key 无效） | 记录日志，触发降级（见 3.3.2.4） |
  | 429 | 限流 | 触发降级或排队重试 |
  | 500 | DeepSeek 服务异常 | 触发降级 |
  | 504 | 网关超时（>10s） | 触发降级 |

- **数据格式**：请求/响应均为 JSON；`message.content` 内嵌 JSON 字符串需二次解析。
- **通信协议**：HTTPS，超时阈值 10 秒，失败重试 1 次后降级。

##### 3.3.2.2 SRS-044 结构化输出（JSON 7 模块）

- **接口标识**：SRS-044，结构化输出，优先级 P0，状态已确认。
- **接口描述**：约定 AI 翻译结果的统一 JSON 结构，包含翻译/关键词/语境/潜台词/建议/风险/相关 7 个模块，供前端按模块渲染。
- **接口图（文字描述）**：DeepSeek `message.content`（JSON 字符串）→ 后端 `json.loads` → 字段校验/补默认值 → 包装为统一响应 `{code,data,msg}` → 前端按模块渲染结果卡片。
- **请求体**：同 3.3.2.1，依赖 `response_format: json_object` 约束输出。
- **响应体（JSON 示例，7 模块完整结构）**：
  ```json
  {
    "code": 0,
    "msg": "ok",
    "data": {
      "translation": "就是"给支持、给资源"的意思，常被滥用成"帮忙"的高级说法。",
      "keywords": ["赋能", "支持", "资源投入"],
      "context": "多见于互联网公司周报、OKR、汇报材料，用"赋能"显得更专业。",
      "subtext": "潜台词往往是"我帮你/我支持你"，但具体怎么帮常常没说清。",
      "suggestion": "日常沟通建议直接说"帮忙""支持"，比"赋能"更清楚。",
      "risk": "low",
      "related": [
        { "id": 12, "word": "抓手" },
        { "id": 28, "word": "对齐" }
      ]
    }
  }
  ```
- **字段说明**：

  | 字段 | 类型 | 说明 |
  |---|---|---|
  | translation | string | "人话"翻译结果，必填 |
  | keywords | string[] | 关键词列表 |
  | context | string | 使用语境说明 |
  | subtext | string | 潜台词/弦外之音 |
  | suggestion | string | 使用建议 |
  | risk | string | 风险等级，枚举 low/medium/high |
  | related | object[] | 相关词条，含 id 与 word |

- **数据格式**：JSON；缺失字段由后端补默认值（空串/空数组/`low`）。
- **通信协议**：HTTPS；前后端通过 `POST /api/translate` 交互，响应即上述结构。
- **错误码**：沿用 3.3.2.1；字段校验失败不抛错，按默认值补全。

##### 3.3.2.4 SRS-046 失败降级（退化为纯词库模式）

- **接口标识**：SRS-046，失败降级，优先级 P1，状态已确认。
- **接口描述**：当 AI 调用失败、超时或限流时，系统自动退化为纯词库模式，从本地词条库匹配释义返回，保证核心可用性。
- **接口图（文字描述）**：`POST /api/translate` → 调 DeepSeek → 失败/超时/限流 → 捕获异常 → 查询本地词条库（按精确匹配/模糊匹配） → 命中则返回基础释义（标记 `fallback: true`） → 未命中返回通用提示。
- **请求 URL / 方法 / 头 / 体**：同 3.3.2.1（降级为后端内部逻辑，前端请求不变）。
- **响应体（JSON 示例，降级返回）**：
  ```json
  {
    "code": 0,
    "msg": "AI 服务暂不可用，已切换词库模式",
    "data": {
      "translation": "（来自词库）赋能：给予支持、提供资源。",
      "keywords": ["赋能"],
      "context": "",
      "subtext": "",
      "suggestion": "",
      "risk": "low",
      "related": [],
      "fallback": true
    }
  }
  ```
- **降级返回说明**：相比正常响应增加 `fallback: true` 标记；`context/subtext/suggestion` 等 AI 生成模块允许为空，前端据此隐藏对应卡片或展示"AI 模块暂不可用"提示。
- **触发条件**：DeepSeek 调用超时（>10s）、返回非 2xx 状态码、429 限流、JSON 解析失败。
- **数据格式**：JSON，结构与 3.3.2.2 兼容（超集）。
- **通信协议**：HTTPS；降级过程对前端透明，仅通过 `fallback` 字段标识。
- **错误码**：降级成功仍返回 `code: 0`；词库亦未命中时返回 `code: 2001, msg: "未找到相关词条"`。

### 3.4 CSCI 内部接口需求

**需求描述：** 本 CSCI 内部接口需求规定"中译中"应用前端客户端与后端服务之间、后端服务与持久化数据库之间的接口约定。系统采用前后端分离架构，前端 Uni-app 客户端与后端 FastAPI 服务之间通过 RESTful API 进行通信，所有接口请求体与响应体均采用 JSON 格式封装，遵循 HTTP 语义规范（GET 查询、POST 创建、PUT 更新、DELETE 删除），统一返回 {code, data, msg} 三段式结构以便于前端解析与异常处理（SRS-047）。后端与持久化层之间通过 SQLAlchemy ORM 访问 SQLite 数据库，所有数据操作通过模型对象完成，禁止直接拼接 SQL，确保接口契约稳定且便于后续向 MySQL 等数据库迁移（SRS-048）。

**具体指标与约束：** 所有 RESTful 接口必须以 /api/v1 为前缀进行版本化管理；请求与响应统一使用 application/json 内容类型；分页查询统一使用 page 与 page_size 参数，响应中返回 total、items 字段；数据库连接采用连接池管理，连接字符串通过环境变量注入，禁止硬编码；ORM 模型必须与数据库表结构一一对应，迁移脚本通过 Alembic 管理。

**合格性规定：** 通过接口文档审查确认所有接口符合 RESTful 规范与 JSON 格式要求；通过接口自动化测试用例验证 {code, data, msg} 结构一致性；通过 ORM 单元测试验证数据库接口的正确性与隔离性；接口契约变更必须同步更新 API 文档并通过回归测试方可发布。

### 3.5 CSCI 数据需求

**需求描述：** 本 CSCI 数据需求定义系统需持久化的核心数据实体及其字段构成。词条数据是系统的核心数据资产，词条表应至少包含以下字段：词、解释、示例、分类、风险等级、来源、状态、创建时间、更新时间，其中分类支持多级分类体系，初始预置 12 个一级分类共约 1200 条词条（SRS-049）。用户数据用于承载用户身份与个性化信息，用户表应至少包含：账号、昵称、偏好设置、学习进度、成就等字段，支持多种登录方式关联同一用户主体（SRS-050）。翻译记录用于留存用户翻译历史以便复盘与归档，翻译历史表应至少包含：原文、翻译结果、翻译时间、所属用户等字段，单用户最近记录保留 100 条（SRS-051）。纠错记录用于支撑词条内容众包修订，纠错表应至少包含：词条ID、纠错内容、提交人、审核状态等字段，与词条表通过外键关联（SRS-052）。

**具体指标与约束：** 所有数据表必须包含主键、创建时间、更新时间三个基础字段；外键关系必须显式声明并设置合理的级联策略；枚举字段（如风险等级、状态）必须在数据字典中明确定义取值范围；时间字段统一使用 UTC 时间存储；初始词条数据应可通过 CSV/JSON 批量导入；用户学习进度与成就字段需支持增量更新。

**合格性规定：** 通过数据模型评审确认表结构与字段完整性；通过数据迁移脚本验证初始数据导入的正确性；通过数据一致性测试验证外键约束与枚举取值的有效性；词条库条目数量与分类覆盖度需达到初始指标要求（1200 条、12 个一级分类）方可验收。

### 3.6 适应性需求

**需求描述：** 本 CSCI 适应性需求描述系统应对数据规模增长与终端形态扩展的能力。在数据适应性方面，词条库应支持可持续扩充，分类体系支持从一级向二级、三级延伸，新增词条与分类不需要变更数据模型与代码结构，仅需通过后台管理或数据导入完成（SRS-053）。在终端适应性方面，系统采用 H5 移动端优先策略，一期交付物为适配移动端浏览器的 H5 应用，后续可在不重写业务逻辑的前提下扩展为微信小程序与原生 App 形态，依托 Uni-app 一套代码多端编译的能力实现终端扩展（SRS-054）。

**具体指标与约束：** 词条库数据量在 10 万条以内时查询响应时间不应明显劣化；分类层级扩展至三级时前端导航与后台管理均应正常工作；H5 应用必须在主流移动端浏览器（Chrome、Safari、微信内置浏览器）中正常显示与交互；响应式布局适配 320px–414px 主流移动分辨率。

**合格性规定：** 通过压力测试验证词条库规模扩展后的查询性能；通过多端编译构建验证 Uni-app 代码的多端适配能力；通过多浏览器兼容性测试验证 H5 应用的适配效果；新增分类层级需在不修改代码的前提下完成配置并通过功能测试。

### 3.7 安全性（Security）需求

**需求描述：** 本 CSCI 安全性需求聚焦内容安全维度。词条必须按低、中、高三级风险进行标记，风险等级作为内容展示与审核策略的依据（SRS-055）；高风险词条仅做客观解释并附加警示说明，不提供鼓励传播的展示形式与交互入口（SRS-056）；用户提交的词条、纠错、投票等内容必须经过"用户提交—管理员审核—用户投票"三段式审核机制后方可公开（SRS-057）。

**具体指标与约束：** 风险等级字段取值限定为 low/medium/high 三级；高风险词条页面必须显示警示横幅且不提供分享按钮；用户提交内容初始状态为"待审核"，审核通过后方可对外可见；管理员审核应在用户提交后 48 小时内处理；用户投票达到阈值的内容可加速进入公开状态。

**合格性规定：** 通过安全扫描与渗透测试验证系统不存在 SQL 注入、XSS 等常见漏洞；通过功能测试验证高风险词条的展示策略与三段式审核流程的执行；通过权限测试验证用户数据隔离的有效性；审核流程各状态流转需通过端到端测试用例验证。

### 3.8 保密性（Safety）需求

**需求描述：** 本 CSCI 保密性需求关注敏感信息保护与输入安全。在密钥保护方面，DeepSeek API 密钥、数据库连接串、JWT 密钥等敏感凭据必须存放于服务端环境变量或密钥管理服务中，严禁以前端硬编码、提交至代码仓库或写入客户端可访问资源的方式暴露（SRS-058）。在输入安全方面，所有来自客户端的输入必须经过校验与过滤，防止 XSS 攻击与 SQL 注入攻击，前端输入需进行 HTML 转义与白名单校验，后端通过 Pydantic 模型校验入参合法性，数据库操作通过 SQLAlchemy ORM 参数化查询执行（SRS-059）。

**具体指标与约束：** 所有密钥必须通过 .env 文件或环境变量注入，.env 文件必须加入 .gitignore；调用第三方 AI 服务时不得传递用户敏感信息（如手机号、账号）；前端展示用户生成内容时必须进行 HTML 转义；后端接口入参必须定义 Pydantic 模型并启用严格校验；上线环境必须强制 HTTPS 传输。

**合格性规定：** 通过代码扫描确认无硬编码密钥；通过 .gitignore 审查确认密钥文件未被纳入版本控制；通过 XSS 与 SQL 注入测试用例验证输入过滤的有效性；通过依赖扫描确认无已知漏洞组件；密钥泄露检查需作为发版前的必检项。

### 3.9 环境需求

**需求描述：** 本 CSCI 环境需求定义系统运行所依赖的服务器与客户端环境。服务器端必须部署于 Linux 操作系统（推荐 Ubuntu 20.04 LTS 或 CentOS 7+），运行 Python 3.10 及以上版本的解释器，依赖 SQLite 3.30 及以上版本的嵌入式数据库，并通过 Nginx 作为反向代理对外提供服务（SRS-060）。客户端环境为支持现代 Web 标准的 H5 浏览器，包括 Chrome 90+、Safari 14+、微信内置浏览器等主流移动端浏览器，无需安装客户端运行时。

**具体指标与约束：** 服务器 Python 版本不得低于 3.10，依赖包通过 requirements.txt 锁定版本；客户端浏览器必须支持 ES2017 语法与 Flexbox 布局；不兼容 IE 浏览器；服务运行依赖的第三方 API（DeepSeek）需可访问；服务器需开放 80/443 端口对外服务。

**合格性规定：** 通过在目标 Linux 环境中完成部署与冒烟测试验证服务器环境要求；通过多浏览器兼容性测试矩阵验证客户端环境要求；依赖清单与部署文档需作为交付物提供并通过新人独立部署验证。

### 3.10 计算机资源需求

**需求描述：** 本 CSCI 计算机资源需求定义系统运行所需的硬件资源与客户端资源。服务器侧最低配置为 2 核 CPU、4GB 内存、20GB 磁盘空间，其中磁盘空间需预留至少 1GB 用于 SQLite 数据库与日志存储，AI 调用产生的网络带宽需满足并发请求要求（SRS-061）。客户端侧资源要求为支持现代 Web 标准的移动端浏览器，浏览器需启用 JavaScript 与 Cookie，可用内存不低于 1GB 以保证页面渲染与交互流畅（SRS-062）。

**具体指标与约束：** 服务器在峰值并发 50 QPS 下 CPU 占用率不应持续超过 80%、内存占用不应超过 3GB；客户端首屏加载资源压缩后不超过 2MB；浏览器需支持 Service Worker 与 LocalStorage 以实现本地缓存；数据库文件增长至 500MB 时应触发归档评估。

**合格性规定：** 通过压力测试验证服务器资源占用符合指标；通过资源监控确认峰值场景下资源使用率不超阈值；通过多机型浏览器兼容性测试验证客户端资源需求；首屏加载性能需通过 Lighthouse 等工具测量并达标。

### 3.11 软件质量特性需求

**需求描述：** 本 CSCI 软件质量特性需求覆盖可靠性与可维护性两个关键质量属性。在可靠性方面，系统应支持 7×24 小时连续运行，核心功能（词条查询、翻译、热词）可用率不低于 95%，单次故障恢复时间不超过 5 分钟，故障不应对已持久化数据造成不可恢复的损失（SRS-063）。在可维护性方面，系统采用前后端分离的模块化设计，业务逻辑按领域模块组织，代码需具备完整的单元测试覆盖，核心模块代码覆盖率不低于 70%，配置与代码分离以便于环境迁移（SRS-064）。

**具体指标与约束：** 核心接口 P99 响应时间不超过 500ms（不含 AI 调用）；页面首屏加载时间不超过 2 秒；后端模块单元测试覆盖率 ≥ 70%；日志需记录足够信息以支撑故障定位；配置项通过环境变量注入；核心功能可用率 ≥ 95%。

**合格性规定：** 通过可用率统计与故障注入演练验证可靠性指标；通过单元测试覆盖率报告验证可维护性指标；通过代码审查确认模块化设计落地；故障恢复演练需在 5 分钟内完成服务恢复方可视为达标。

### 3.12 设计和实现约束

**需求描述：** 本 CSCI 设计和实现约束定义项目技术栈选型与开源组件复用策略。技术栈方面，前端采用 Uni-app（基于 Vue 3 语法）开发 H5 应用，后端采用 Python + FastAPI 框架提供 RESTful API，数据持久化采用 SQLite 嵌入式数据库，AI 能力通过调用 DeepSeek API 实现，系统遵循前后端分离架构，前端通过 HTTP API 调用后端服务（SRS-065）。开源组件复用方面，前端 UI 组件采用 uView UI，后端用户认证与权限管理复用 FastAPI Users，后台管理系统基于 Vue-element-admin 框架二次开发，通过复用成熟开源组件降低开发成本与风险（SRS-066）。

**具体指标与约束：** 前端必须基于 Uni-app + Vue 3，不得引入与 Uni-app 不兼容的依赖；后端必须基于 FastAPI，依赖管理使用 pip + requirements.txt；数据库必须通过 SQLAlchemy ORM 访问，不直接拼接 SQL；开源组件必须遵循其开源协议，二开代码需保留原始版权声明；AI 调用必须封装为独立服务层以便替换模型。

**合格性规定：** 通过技术栈审查确认技术选型符合约束；通过依赖清单审查确认开源组件协议合规；通过架构评审确认前后端分离与模块化设计落地；技术栈变更需经评审并更新本文档。

### 3.13 人员需求

**需求描述：** 本 CSCI 人员需求定义系统涉及的用户角色及其权限边界。系统用户分为三类角色：普通用户（注册登录后可使用词条查询、翻译、投票、纠错、收藏、学习等完整功能）、管理员（在普通用户权限基础上具备后台管理、词条审核、数据导入导出、用户管理等管理权限）、游客（未登录用户可浏览部分公开内容但不可提交内容、投票或进行个性化操作）（SRS-067）。

**具体指标与约束：** 所有角色权限必须基于角色访问控制（RBAC）模型实现；游客可访问的内容范围由后端接口白名单控制；管理员账号必须支持双因素认证；角色与权限映射关系必须可配置；不同用户之间的数据必须按用户身份进行隔离。

**合格性规定：** 通过权限矩阵测试验证各角色可访问功能符合定义；通过越权访问测试验证游客与普通用户无法访问管理接口；通过角色配置验证权限可配置性；管理员双因素认证必须启用并通过测试。

### 3.14 培训需求

**需求描述：** 本 CSCI 培训需求聚焦用户上手引导。系统必须提供面向新用户的三步引导流程，引导用户依次完成"翻译一个词—浏览一个词条—收藏或投票"三个核心操作，引导流程在用户首次进入应用时自动触发，支持跳过与再次触发，引导完成后用户应能独立使用核心功能而无需查阅帮助文档（SRS-068）。

**具体指标与约束：** 引导流程必须在 3 步内完成；每步引导必须包含可视化高亮与简短文字说明；引导必须支持"跳过"与"不再提示"选项；引导状态需持久化到 LocalStorage 以避免重复触发；用户可通过设置入口手动重新触发引导。

**合格性规定：** 通过新用户可用性测试验证三步引导流程的有效性；通过功能测试验证引导的跳过、再次触发与状态持久化；引导流程完成率应不低于 80% 方可验收。

### 3.15 后勤保障需求

**需求描述：** 本 CSCI 后勤保障需求定义系统部署与运维保障方式。系统采用 Docker 容器化部署，前后端分别构建为独立镜像，通过 docker-compose 编排启动，对外通过 Nginx 反向代理统一暴露端口并处理 HTTPS 证书；数据库每日自动备份并保留近 7 天备份；AI 调用额度需监控并设置阈值告警（SRS-069）。

**具体指标与约束：** 必须提供 Dockerfile 与 docker-compose.yml；Nginx 配置必须启用 Gzip 压缩与静态资源缓存；数据库每日凌晨自动备份至独立目录并保留近 7 天；AI 调用量接近额度阈值（80%）时必须触发告警；部署文档必须作为交付物提供。

**合格性规定：** 通过在目标环境中执行 docker-compose up 完成部署并冒烟测试验证部署方式；通过模拟备份恢复演练验证数据备份的有效性；通过额度监控演练验证告警机制；部署文档需经评审并通过新人独立部署验证。

### 3.16 其他需求

**需求描述：** 本 CSCI 其他需求涵盖国际化预留与无障碍支持。在国际化方面，系统当前仅提供简体中文界面，但代码层面必须预留 i18n 国际化接口，文案集中管理于语言资源文件中，便于后续扩展多语言版本（SRS-070）。在无障碍方面，系统必须提供基础无障碍支持，包括字号可调（至少三档）、对比度可调（深色/浅色模式），关键交互元素需提供 ARIA 标签以兼容辅助技术（SRS-071）。

**具体指标与约束：** 所有界面文案必须通过 i18n 资源文件引用，禁止硬编码中文；字号调节必须覆盖正文与标题；深色模式必须覆盖全部页面；图标按钮必须提供 aria-label 属性；版本号遵循语义化版本规范，每次更新附带 CHANGELOG。

**合格性规定：** 通过代码审查确认文案均通过 i18n 接口引用；通过无障碍扫描工具（如 axe）验证关键页面无严重违规；通过深色模式全页面浏览验证覆盖率；字号调节功能需在所有正文区域生效并通过测试。

### 3.17 包装需求

**需求描述：** 本 CSCI 包装需求定义项目交付物清单。最终交付物必须包含：可访问的 H5 应用（通过 URL 直接访问）、后端服务（Docker 镜像与部署脚本）、后台管理系统（独立前端应用）、API 文档（基于 OpenAPI/Swagger 自动生成）、GJB 格式需求与设计文档（含需求规格说明、设计说明等）（SRS-072）。

**具体指标与约束：** H5 应用必须可通过公网 URL 直接访问且无需安装；后端服务必须以 Docker 镜像形式交付并附 docker-compose.yml；API 文档必须由代码注解自动生成并与代码同步；GJB 文档必须符合 GJB 438C-2021 格式要求；交付物需提供版本号与构建清单。

**合格性规定：** 通过交付物清单核对确认所有交付物齐全；通过 H5 应用 URL 访问验证可用性；通过 API 文档与代码一致性检查验证文档准确性；GJB 文档需经格式审查确认符合标准。

### 3.18 需求的优先顺序和关键程度

**需求描述：** 本 CSCI 需求的优先顺序和关键程度定义所有需求的优先级体系与关键程度判定规则。优先级分为四级：P0（核心需求，必须实现，缺失则系统不完整）、P1（重要需求，强烈建议实现，缺失则影响核心体验）、P2（一般需求，有了更好，缺失不影响核心体验）、P3（锦上添花需求，可在后续迭代中实现）（SRS-073）。关键程度判定规则为：所有 P0 级需求均为关键需求，必须 100% 实现并通过验收，任何 P0 需求未达成均视为项目验收不通过；P1 级需求为重要需求，应尽量实现，缺失需在遗留问题中说明；P2、P3 级需求为非关键需求，可根据资源情况调整（SRS-074）。

**具体指标与约束：** 每条需求必须明确标注优先级（P0/P1/P2/P3）；P0 需求覆盖率必须达到 100% 方可验收；需求优先级变更必须经评审并更新本文档；关键需求清单需在需求追踪矩阵中独立标注；P1 需求实现率应不低于 80%。

**合格性规定：** 通过需求清单审查确认每条需求均标注优先级；通过 P0 需求覆盖率统计验证关键需求 100% 实现；通过需求追踪矩阵验证关键需求与设计、测试用例的双向追踪关系；P0 需求未达成项必须在遗留问题清单中记录并说明影响。

### 4.3 合格性判定准则

| 准则类别 | 判定要求 |
|---|---|
| P0 需求合格判定 | 所有 P0 级需求必须 100% 通过其指定的全部验证方法，否则视为项目验收不通过。 |
| P1 需求合格判定 | P1 级需求实现率应不低于 80%，未达成项必须在遗留问题清单中记录并说明影响。 |
| P2/P3 需求合格判定 | P2、P3 级需求根据资源情况实现，未实现项在遗留问题清单中记录。 |
| 多方法需求合格判定 | 一条需求指定多种验证方法时，所有方法均需通过方视为该需求合格。 |
| 遗留问题处理 | P0 需求未达成项必须在遗留问题清单中记录并说明影响范围与解决计划。 |

### 7.3 优先级说明

本 SRS 中所有需求均标注优先级，优先级分为 P0、P1、P2、P3 四级，定义如下：

| 优先级 | 定义 | 验收要求 |
|---|---|---|
| P0 | 核心需求，必须实现，缺失则系统不完整或核心价值无法体现。 | 覆盖率必须达到 100% 方可验收，任何 P0 需求未达成均视为项目验收不通过。 |
| P1 | 重要需求，强烈建议实现，缺失则影响核心体验或评审关键维度。 | 实现率应不低于 80%，未达成项必须在遗留问题清单中记录并说明影响。 |
| P2 | 一般需求，有了更好，缺失不影响核心体验，可在资源允许时实现。 | 根据资源情况实现，未实现项在遗留问题清单中记录。 |
| P3 | 锦上添花需求，可在后续迭代中实现，本期不强制要求。 | 根据资源情况实现，未实现项在遗留问题清单中记录。 |

优先级变更须经过评审并同步更新本文档与需求追踪矩阵。

### 7.4 需求状态说明

本 SRS 中需求状态分为以下四类，用于跟踪需求从提出到确认的生命周期：

| 状态 | 定义 | 后续动作 |
|---|---|---|
| 待确认 | 需求刚提出，尚未经过讨论与评审，存在不确定性。 | 进入需求评审会议讨论，明确是否纳入及优先级。 |
| 已确认 | 需求经过讨论与评审，确定要做，已分配优先级与 SDD 对应章节。 | 进入设计与开发流程，纳入需求追踪矩阵。 |

| 待调整 | 需求有修改意见，需要重写或重新定义后再次评审。 | 修订需求描述后重新进入评审流程。 |
